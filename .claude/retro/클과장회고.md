## 2026-04-17 — PlaceSelectionPage 재설계 (제대리 리뷰 반영)

### 무엇을 틀렸나
- `loadPlaces` 내부에 `REGION_AREA_CODE` 맵을 하드코딩했다. `tourApi.ts`에 이미 `REGION_AREA_CODES`가 export되어 있었는데 확인하지 않고 중복 생성함.
- `parseInt(subItem.subnum)`을 그대로 사용했다. 빈 문자열이나 예상 외 포맷일 경우 NaN이 되어 정렬 로직이 깨질 수 있었음.

### 왜 틀렸나
- 새 로직 작성에 집중하다 기존 상수/유틸 먼저 확인하는 습관이 빠졌다.
- `parseInt` vs `Number` 차이를 알면서도 "빠르게 쓰자"는 판단이 앞섰다.

### 다음엔 어떻게 할 건가
- 지역 코드처럼 다른 파일에서 쓰일 법한 상수를 작성하기 전에 `tourApi.ts`, `constants` 등 기존 파일을 grep 먼저 한다.
- 문자열→숫자 변환은 항상 `Number(...) || 0` 패턴으로 기본값 포함. `parseInt`는 명시적 기수(radix)가 필요한 경우에만.

## 2026-04-18 — push 누락 + 관광공사 추천 섹션 구조 버그

### 무엇을 틀렸나
1. 커밋 후 push를 안 해서 Vercel 배포가 2커밋 뒤쳐진 상태로 후추님 QA 진행됨
2. 관광공사 추천 섹션 설계 시 subItems의 subcontentid가 우리가 로드한 places에 없다는 걸 처음에 파악 못함. matchedRecommendCount 체크만 추가하고 실제 장소 fetch는 빠뜨림

### 왜 틀렸나
1. 커밋 직후 배포 확인 습관이 없었음 — "커밋=배포"로 착각
2. 추천 섹션 설계 시 API 두 단계 구조(코스→subcontentid→장소상세)를 온전히 그리지 않고 매칭 체크만 추가함

### 다음엔 어떻게 할 건가
- 커밋 후 반드시 push 확인. 배포 URL 반영 여부까지 확인하고 보고
- API 연동 시 "어떤 ID로 어떤 데이터를 join하는지" 전체 데이터 흐름 그려보고 구현

## 2026-04-17 오후 — 리뷰 재검증 및 Critical 수정

### 무엇을 틀렸나
1. 오전 리뷰 작성 시 Gemini 모델명 `gemini-2.5-flash`를 "v1beta 미지원"으로 오판정. 실제로는 정상 동작하는 검증된 모델이었음 (메모리 `feedback_gemini_model_validation.md`에 이미 기록됨)
2. 서브에이전트(Explore)가 리턴한 "gemini-2.5-flash doesn't exist" 단언을 그대로 수용. 메모리 교차확인 없이 리뷰 문서에 그대로 옮김
3. 제대리도 같은 판정을 냈지만 이건 grounding 검색 결과와 실제 API 동작의 괴리. 제대리 답변도 무조건 신뢰하면 안 됨

### 왜 틀렸나
- 오전 EatScan 세션에서 같은 모델을 수십 번 호출 검증한 결과를 메모리에 저장해뒀는데, IdalTrip 리뷰 쓸 때 해당 메모리를 안 꺼냄
- 서브에이전트 리포트와 메모리가 충돌할 때 메모리(실동작 검증)를 우선해야 함을 놓침

### 다음엔 어떻게 할 건가
- Gemini/외부 API 모델명 언급되면 반드시 관련 메모리 먼저 확인 후 판정
- 서브에이전트/제대리가 "모델이 없다/동작 안 한다"라고 하면 실제 API 호출로 1회 검증 후 리뷰에 반영
- 리뷰 재검증 시 과거 메모리 + 실행 검증 > 에이전트 단언
