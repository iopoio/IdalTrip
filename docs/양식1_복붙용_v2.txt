1) 서비스 기획배경 및 필요성
--------------------------------------------------------------------------------

[1. 서비스 기획 배경]

여행을 떠나기 전, 가장 먼저 마주치는 건 풍경이 아니라 '결정 피로'입니다. 어디로 갈지, 언제가 적기인지, 무엇을 할지 — 평균적으로 사람들은 주말 여행 하나를 정하는 데 3~5시간을 검색에 씁니다. 네이버에서 축제를 찾고, 지도에서 관광지를 찍어보고, 블로그에서 맛집을 검증하고, 카카오맵에서 동선을 그립니다. 이 과정에서 정작 여행의 즐거움보다 '계획의 부담'이 더 크게 다가옵니다.

한국관광공사가 축제·관광지·음식점·숙박 데이터를 OpenAPI로 공개하고 있지만, 일반 사용자가 이를 직접 활용하긴 어렵다. 데이터는 풍부한데 "이번 달 나에게 맞는 여행" 이라는 맥락으로 묶이지 않기 때문입니다.

'이달여행'은 이 간극을 좁힙니다. 사용자가 입력하고 검색하고 비교할 필요 없이, 진입 즉시 "이번 달, 지금 갈 만한 곳" 을 자동 큐레이션합니다. 한국관광공사 TourAPI 4.0을 기반으로 시간(월별 축제)과 공간(6권역) 두 축을 교차하여, "이번 달 어디 갈까?" 라는 단 한 번의 고민조차 줄여줍니다.

[2. 서비스 필요성]

- 현대인 의사결정 피로의 해소: 정보 과잉의 시대, 사용자가 가장 원하는 건 더 많은 데이터가 아니라 '지금 나에게 맞는 한 가지 답'입니다. 이달여행은 진입 즉시 큐레이션을 노출하여 의사결정 비용을 0에 가깝게 낮춥니다.
- 공공 관광데이터의 사용자 체감 확대: TourAPI의 풍부한 데이터가 B2B·연구용에 머무르지 않고 일반 사용자가 매월 자연스럽게 접하는 서비스로 재구성될 필요가 있습니다. 축제·관광지·음식점을 한 화면에서 시간축으로 큐레이션해 데이터 활용도를 체감 가능한 수준으로 끌어올립니다.
- 국내 여행 실행 전환율 제고: 시즌 축제는 지역경제 활성화의 핵심 자산입니다. 사용자가 "내가 가면 이런 일정이 나온다"를 즉시 확인하면 결정에서 실행 사이의 마찰이 줄어들어 실제 방문 전환이 극대화됩니다.

2) 서비스 개요
--------------------------------------------------------------------------------

[1. 기획 서비스 소개]
"고민하지 마세요. 이번 달은 여기 가세요." — 시즌 축제 기반 AI 여행 코스 자동 큐레이션 모바일 웹

[2. 기획 서비스 주요 기능]
1. 진입 즉시 시즌 축제 피드: 사용자가 별도 입력 없이 사이트에 진입하면 이번 달의 축제를 전국·권역별로 자동 큐레이션
2. 축제 상세 + 주변 탐색: 선택한 축제 주변 관광지/맛집을 반경 기반으로 즉시 제시
3. AI 코스 자동 생성: 선택한 장소들을 Gemini AI가 이동시간·체류시간 고려해 하루 또는 1박2일 타임라인으로 조합
4. 경량 경로 엔진: 자체 하버사인 연산으로 외부 API 지연 없이 즉시 코스 렌더링
5. 카카오맵 내비 직연결: 완성된 코스에서 각 장소를 카카오맵 공식 URL Scheme으로 바로 길안내 연결

[3. 서비스 차별성]

(1) 진입 즉시 가치 제공 (Zero-input Value)
대부분의 여행 정보 서비스는 사용자가 지역·기간·테마를 입력하고 검색해야 결과를 받습니다. 이달여행은 사용자의 별도 입력 없이 사이트 진입 순간 '이번 달 갈 만한 곳' 큐레이션을 즉시 노출합니다. 의사결정 피로가 큰 현대인에게 "고민할 필요 없이 바로 보여주는" 진입 경험은 강력한 차별 가치입니다.

(2) 시간축 × 공간축 2D 큐레이션
- 공간 중심 서비스: 지역 맛집/관광지 정보는 풍부하나 '언제'라는 시간적 맥락이 부족합니다.
- 시간 중심 서비스: 단순 행사 달력 나열에 그쳐 지역별 동선 설계가 어렵습니다.
- 이달여행: 월별(시간) × 6권역(공간)의 교차 큐레이션과 AI 동선 완성을 결합하여 "이번 달 어디를, 어떻게" 갈지에 대한 답을 한 번에 제시합니다.

(3) 기술적 우위 및 확장성
- 공개 API 생태계 기반 재설계: OpenStreetMap 기반 지도 렌더링과 카카오맵 공식 URL Scheme 연동으로 외부 API 토큰 의존도를 최소화하여 운영 안정성을 확보했습니다.
- 즉시 응답성 확보: 자체 하버사인(Haversine) 연산 로직을 통해 네트워크 지연 없이 코스를 즉시 렌더링하여 모바일 환경에 최적화된 UX를 제공합니다.

[4. 지역 특화 관련 사항]
본 서비스는 전국 단위를 대상으로 하되 6권역(서울·경기, 강원, 충청, 전라, 경상, 제주)별 섹션 구조를 적용합니다. 각 권역의 TourAPI areaCode를 체계적으로 매핑하여 지역성 있는 큐레이션을 제공하며, 향후 강원·제주 등 축제 밀도가 높은 권역에 대한 심화 큐레이션을 우선적으로 구축할 계획입니다.

3) 데이터 활용 방안
--------------------------------------------------------------------------------

[1. 활용 예정 한국관광공사 OpenAPI] (TourAPI 4.0 / KorService2 기준)
- 행사정보조회 (searchFestival2): 월별·지역별 축제 피드 생성
- 키워드검색 (searchKeyword2): 축제 주변 관광지/맛집 탐색
- 공통정보조회 (detailCommon2): 장소 상세 정보(개요, 이미지) 로드
- 위치기반관광정보 (locationBasedList2): 현재 위치 및 축제 좌표 기반 반경 탐색

[2. 데이터 활용 방식]

(1) 진입 → 자동 큐레이션 → 매시업 흐름
1. 사용자 사이트 진입 시 현재 월·전국 기준으로 'searchFestival2'를 자동 호출하여 이번 달 축제 피드를 즉시 노출합니다.
2. 축제 선택 시 해당 좌표를 키로 'searchKeyword2' 및 'locationBasedList2'를 호출하여 주변 관광지와 음식점 정보를 수집합니다.
3. 선택된 장소들의 'detailCommon2' 정보를 로드하여 카드 UI를 구성하고, 이를 Gemini AI 프롬프트에 전달하여 최적의 타임라인(JSON)을 생성합니다.

(2) 데이터 정합성 및 운영 안정성
- 날짜 범위 필터링: TourAPI의 행사 시작/종료일 데이터를 기반으로 종료된 축제를 실시간으로 자동 제외합니다.
- 상수 기반 관리: areaCode 매핑 테이블을 프런트엔드 상수로 관리하여 데이터 정합성을 유지하고 확장성을 높였습니다.
- 폴백(Fallback) 구조: 이미지 누락이나 데이터 공백 발생 시 기본 이미지 및 안내 텍스트가 표시되도록 방어 설계되었습니다.

4) 서비스 발전 방향
--------------------------------------------------------------------------------

[1. 개발 서비스 향후 발전방향]

- 단기: 지역 RTO(Regional Tourism Organization)와의 협력을 통해 공식 축제 데이터의 검증 루프를 구축하고, Gemini 프롬프트 튜닝을 통해 코스 생성 품질을 지속적으로 고도화하겠습니다.
- 중기: OSRM(OpenRouteService) 연동을 추가하여 실제 도로 기반 이동 시간을 반영하고, 숙박 데이터(TourAPI 32번)를 통합하여 1박 2일 이상의 장기 코스 추천으로 확장하겠습니다. 또한 공식 매거진(KTX 매거진 등) 및 축제 상세 콘텐츠 채널과의 협력을 통해 OpenAPI 외 데이터(프로그램 일정·주변 정보·검증된 큐레이션 등)를 통합함으로써 매거진 라이프사이클 연장과 코스 품질 강화를 동시에 달성하는 윈윈 모델을 검토하고, 다국어 TourAPI를 연동하여 인바운드 관광객 시장도 공략할 예정입니다.
- 장기: AI 코스 생성 엔진의 B2B API 라이선싱을 추진하고, 지자체와의 제휴를 통해 '이달여행 공식 등재 축제' 인증 시스템을 도입하여 지역 관광 활성화의 선순환 구조를 만들겠습니다.

[2. 서비스 지속 가능성]
- Vercel Edge 네트워크 기반의 서버리스 아키텍처를 채택하여 트래픽 증가에 따른 운영 비용을 극소화했습니다.
- 공공 OpenAPI 기반의 데이터 수집으로 원천 데이터 비용이 발생하지 않으며, 자동화된 파이프라인을 통해 1인 운영이 가능한 저비용·고효율 구조를 확립했습니다.
